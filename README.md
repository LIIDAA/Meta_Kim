![Method](https://img.shields.io/badge/method-%E6%84%8F%E5%9B%BE%E6%94%BE%E5%A4%A7-0f766e)

**Meta_Kim 是一套跨 Claude Code、Codex、OpenClaw 的元架构工程。**

它不解决“怎么多写几个 agent”。

它解决的是更根本的问题：

**怎么让 AI 不急着回答，而是先把人的原始意图看清、补全、放大，再开始做事。**

换句话说，Meta_Kim 想立的不是一个“prompt 模板仓”，而是一套规矩：

- 用户说的是原始意图，不是完整任务
- 系统不能直接糊答案，必须先做意图放大
- 放大不是废话变多，而是把缺失的目标、约束、风险、路径补出来
- 这个规矩不能只在一个软件里成立，而要同时在 Claude Code、Codex、OpenClaw 里成立

这就是 Meta_Kim 的核心。

* * *

## 它到底是什么

Meta_Kim 不是聊天产品，不是网页，不是 SaaS，也不是一个单独的超级 prompt。

它更像一套“AI 助手内核工程”：

- 有方法论
- 有角色分工
- 有运行时适配
- 有同步机制
- 有校验机制

目标很明确：

**把“基于元的意图放大”做成一套可以真正落到不同 AI 运行时里的工程资产。**

所以这个仓库里你看到的，不只是 prompt。

你会看到：

- agent
- skill
- MCP
- hook
- memory
- workspace
- 配置模板
- 校验脚本

因为 Meta_Kim 要做的不是“让模型多说两句”，而是“让整个 AI 助手系统按同一套方法做事”。

## 它想解决什么问题

现在大多数 AI 助手都有同一个毛病：

**用户只说了半句话，系统就开始给完整答案。**

这会带来三个直接后果：

1. 目标没搞清楚，答案再完整也是偏的。
2. 约束没补出来，执行时迟早返工。
3. 同一个需求换一个软件，结果风格、质量、结构全变样。

Meta_Kim 对这个问题的态度很硬：

**先放大意图，再进入执行。**

也就是先把这些东西补出来：

- 真实目标
- 范围边界
- 风险约束
- 受众对象
- 成功标准
- 交付形态
- 执行顺序

补完这些，AI 才算真正“理解任务”。

## 什么叫“意图放大”

“意图放大”不是修辞动作，而是任务工程动作。

它的意思是：

**把一句模糊、压缩、缺上下文的话，扩展成一个可执行、可判断、可协作的任务结构。**

举个最简单的例子。

用户说：

> 帮我做一个项目。

在普通助手那里，这句话经常会直接得到一份模板化方案。

但在 Meta_Kim 这里，这句话还远远不够，因为它至少缺这些问题：

- 你做给谁用
- 你现在做到哪一步
- 你想解决什么核心问题
- 你最在意速度、质量还是扩展性
- 最终交付是代码、文档、架构还是产品

所以 Meta_Kim 的第一步不是回答，而是把这些缺口补出来。

这就叫意图放大。

## 什么叫“元”

Meta_Kim 里的“元”，不是装饰词，不是玄学词，也不是为了起名字显得特别。

这里的定义很直接：

**元 = 为了完成意图放大而存在的最小可治理单元。**

这句话里有四个重点：

- 最小：它不是无限膨胀的大总管
- 可治理：它有边界，有责任，有输入输出
- 单元：它可以独立存在，也可以参与协作
- 为了意图放大：它不是为了堆角色，而是为了解决任务理解和执行质量问题

所以 Meta_Kim 不是“一个万能 agent”，而是一组有明确职责的元。

这种拆法的意义在于：

- 每个元知道自己该做什么
- 出问题时知道该查谁
- 需要增强时知道该补哪一层
- 不同运行时可以映射同一组职责，而不是各写各的

## 它和普通 prompt 仓库有什么区别

很多仓库会把一堆 prompt、几个 agent 文件、一些工具配置放在一起，然后说自己是“多 agent 系统”。

Meta_Kim 不想做那个方向。

它和普通仓库的差别主要有四点：

1. 它不是 prompt 展示柜。
它不追求“这里有多少个角色”，而追求“这些角色能不能形成稳定治理”。

2. 它不是单运行时方案。
它不是只在 Claude Code 里成立，也不是换到 Codex / OpenClaw 就失效。

3. 它不是把所有能力塞进一个总 prompt。
它强调角色边界、能力分层、运行时映射，而不是让一个 prompt 兼任所有事。

4. 它不是只讲理论。
它把理论做成了可以同步、验证、落地的工程资产。

一句话说：

**普通仓库常常是在“堆能力”，Meta_Kim 要做的是“立规矩”。**

## 这个项目最后要做成什么样

最终效果不是“你看到 8 个角色，很热闹”。

最终效果应该是：

1. 用户提出原始需求
2. 系统先做意图放大
3. 判断是否需要分工
4. 必要时调用不同元 agent
5. 汇总结果，形成更完整的最终输出

所以用户真正感受到的应该是：

- 这个系统不急着答
- 它更会抓重点
- 它更少漏条件
- 它更少中途跑偏
- 它交付出来的东西更像真正能落地的方案

这才是 Meta_Kim 要追求的“牛逼”，不是名字多，而是结果稳。

## 8 个元 agent 分工

这 8 个元 agent 是后台组织结构，不是前台菜单。

- `meta-warden`：统一入口、统筹、仲裁、最终整合
- `meta-genesis`：人格、提示词、`SOUL.md`
- `meta-artisan`：skill、MCP、工具能力匹配
- `meta-sentinel`：hook、安全、权限、回滚
- `meta-librarian`：记忆、知识、连续性
- `meta-conductor`：工作流、节奏、编排
- `meta-prism`：质量审查、漂移检测、反 AI 套话
- `meta-scout`：外部工具发现与评估

如果你第一次接触这个项目，可以先只记一个：

**`meta-warden` 是总入口。**

剩下 7 个元 agent 是它背后的专员体系。

## 三端怎么对应

Meta_Kim 的难点不在“把一套东西写出来”，而在“让三套运行时都遵守同一套规矩”。

| 运行时 | 用户看到的入口 | 仓库里的主要落点 | 作用 |
| --- | --- | --- | --- |
| Claude Code | `CLAUDE.md` | `.claude/`、`.mcp.json` | 让 Claude Code 按 Meta_Kim 的元职责和约束运行 |
| Codex | `AGENTS.md` | `.codex/`、`.agents/`、`codex/config.toml.example` | 让 Codex 使用同一套元结构和技能映射 |
| OpenClaw | `openclaw/workspaces/` | `openclaw/` | 让 OpenClaw 的本地 workspace agent 也按同一套方法工作 |

重点不是三家长得一样。

重点是：

**三家的入口不同，但底层治理目标一致。**

## 仓库怎么读

如果你第一次打开这个仓库，建议按下面的顺序理解：

1. 先看这份 `README.md`
作用：理解项目立场、目标、方法、结构。

2. 再看 `CLAUDE.md` 和 `AGENTS.md`
作用：理解 Claude Code 和 Codex 各自怎么接这个体系。

3. 再看 `.claude/agents/`
作用：看 8 个元 agent 的职责定义。

4. 最后再看 `openclaw/`、`.codex/`、`.agents/`
作用：理解这些职责是怎么被映射到不同运行时里的。

如果你一上来就钻脚本和配置文件，很容易觉得乱。

正确读法是：

**先理解它想立什么规矩，再看它怎么把规矩落地。**

## 仓库结构

```text
Meta_Kim/
├─ .claude/        Claude Code 主源，包括 agents、skills、hooks、settings
├─ .codex/         Codex 会直接读取的仓库内 agents 与 skills
├─ .agents/        Codex 项目级 skills 目录
├─ codex/          只放 Codex 全局配置示例，不是另一套运行时
├─ openclaw/       OpenClaw workspace、模板配置、运行时镜像
├─ scripts/        同步、校验、MCP、自检、OpenClaw 本地准备脚本
├─ shared-skills/  跨运行时共享的技能镜像
├─ AGENTS.md       Codex / 通用运行时入口说明
├─ CLAUDE.md       Claude Code 入口说明
├─ .mcp.json       Claude Code 项目级 MCP 配置
└─ README.md       项目总说明
```

### 为什么会有 `codex/`

这是最容易让人误会的地方。

原因很简单：Codex 的配置分成两部分。

- 一部分是仓库内资产，所以放在 `.codex/` 和 `.agents/`
- 一部分是用户自己电脑里的全局配置，所以仓库里只能给你一个示例文件

因此：

- `.codex/` 是 Codex 真正会直接读的仓库内内容
- `codex/` 只是一个配置示例目录，用来告诉你怎么写 `~/.codex/config.toml`

它不是重复目录，也不是偏心 Codex。

只是 Codex 的配置模型和另外两家不一样。

## 这些命令到底什么时候才需要跑

这部分必须说人话。

不是每个下载这个仓库的人都需要把所有命令跑一遍。

### `npm install`

作用：安装这个仓库依赖的 Node 包。

什么时候需要：
- 你第一次把项目下载到本地，准备真正使用或验证它

什么时候不需要：
- 你只是看文档
- 你只是改一两段纯文字

### `npm run sync:runtimes`

作用：把主源同步成三端真正要吃的运行时文件。

你可以把它理解成“重新生成三端产物”。

什么时候需要：
- 你改了 agent
- 你改了 skill
- 你改了运行时配置
- 你想确保 Claude Code / Codex / OpenClaw 三端资产重新对齐

什么时候不需要：
- 你只改了 README
- 你只改了许可证

### `npm run prepare:openclaw-local`

作用：给 OpenClaw 做本机准备。

OpenClaw 和另外两家不一样，它除了读仓库文件，还依赖你电脑用户目录下的本地授权和 agent 状态。

什么时候需要：
- 你准备在自己的电脑上真正跑 OpenClaw

什么时候不需要：
- 你不用 OpenClaw
- 你只是看项目结构

### `npm run verify:all`

作用：做总验收。

它会统一检查三端资产有没有漏、有没有坏、能不能对上。

什么时候需要：
- 你准备提交
- 你准备发布
- 你准备开源
- 你改了一批运行时资产，想确认没配坏

什么时候不需要：
- 你只是看说明
- 你只是小改一段文档

## 最简单的使用方式

### 如果你只是第一次看项目

你什么命令都不用跑。

先把 README 看完，再去看 `CLAUDE.md`、`AGENTS.md`、`.claude/agents/`。

### 如果你是第一次把它拉到本地，想验证它是活的

在仓库根目录执行：

```bash
npm install
npm run sync:runtimes
npm run verify:all
```

### 如果你还要在本机跑 OpenClaw

再补一条：

```bash
npm run prepare:openclaw-local
```
